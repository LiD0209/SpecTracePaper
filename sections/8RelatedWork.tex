\section{Related Work}
\label{sec:related}

\textbf{Protocol conformance testing and fuzzing.}
Recent work has begun to reduce manual oracle construction by deriving analysis targets from protocol standards. 
RIBDetector extracts RFC-guided rules to find implementation inconsistencies~\cite{chen2022ribdetector}. 
ParCleanse uses LLMs to translate RFC documents into protocol message specifications that act as traceable quasi-oracles for parser validation~\cite{zheng2025parcleanse}. 
RFCAudit builds semantic indexes over protocol code and uses an LLM agent to retrieve implementation context for RFC-guided bug detection~\cite{zheng2025rfcaudit}. 
ProtocolGuard is another recent specification-guided system that uses LLM-guided slicing and assertion-guided fuzzing for protocol non-compliance detection~\cite{song2026protocolguard}.
These systems are closest to \tool{} in that they start from standards rather than relying only on manually written tests. 
However, their semantic anchors differ. 
RIBDetector is organized around RFC-derived rules, ProtocolGuard around packet-processing handlers, message-flow slices, and generated packet-level counterexamples; ParCleanse focuses on packet or message-format constraints; and RFCAudit focuses on repository-level semantic retrieval. 
\tool{} instead uses a document-derived field space to organize the correspondence between standards and code.
It represents each checkable requirement as a field-level rule, and requires each report to identify the normative standard span, the constrained protocol field, the corresponding implementation variable or code context, and the evidence supporting this correspondence.
Thus, tests in \tool{} are not standalone packet-level or parser-level oracles; they validate whether a specific field-level obligation is correctly enforced by the implementation.
When the available code context or runtime behavior is insufficient to support a judgment, \tool{} returns \textsc{Insufficient} rather than forcing a binary violation or non-violation decision.
In our evaluation, this field-centric workflow produced 204 submitted conformance-deviation reports, of which 126 were maintainer-confirmed or fixed, with 0.93 adjudicated precision.
To our knowledge, this is among the largest externally validated result sets reported by recent RFC/spec-guided conformance-checking systems.

Traditional conformance suites and protocol fuzzers provide strong executable evidence, but their oracles are usually hand-written, inferred from observed behavior, or tied to protocol-specific harnesses. 
TLS-Attacker and TLS-Anvil generate targeted TLS tests~\cite{somorovsky2016tlsattacker,maehren2022tlsanvil}, while protocol state fuzzing and automata-based analyses expose deviations in TLS and DTLS state machines~\cite{de2015protocol,fiterau2020dtls,fiterau2023automata,beurdouche2015messy}. 
Certificate-validation studies use adversarial, differential, symbolic, and RFC-directed inputs to test SSL/TLS implementations~\cite{brubaker2014frankencert,chen2015guided,chen2018rfc,chen2017hvlearn,chau2017symcerts}. 
General protocol fuzzers such as AFLNet and StateAFL improve stateful exploration for network protocols~\cite{pham2020aflnet,natella2022stateafl,natella2021profuzzbench,ba2022stateful}. 
These techniques are effective when the target behavior and oracle are already encoded. 
\tool{} is complementary: it starts from normative standard text, derives field-centric oracles, aligns them to implementation evidence, and preserves the standard--rule--code--test trace needed to explain why an observed behavior is a conformance violation rather than merely a behavioral anomaly.

\textbf{Formal models and specification extraction.}
Formal verification provides much stronger guarantees when a precise model is
available. Verified TLS implementations and symbolic protocol analyzers such as
miTLS, ProVerif, Tamarin, AVISPA, and TLS~1.3 models reason about
well-specified security properties~\cite{bhargavan2013mitls,blanchet2016proverif,meier2013tamarin,armando2005avispa,cremers2017tls13}.
Other work extracts protocol state machines from specification documents to
support attack synthesis~\cite{pacheco2022automated}. These approaches either
require a formal model or target state-machine structure. \tool{} makes a
lighter claim: it does not prove whole-protocol correctness, but checks
implementation behavior against normalized field-level requirements that can be
grounded in source code and validated with tests.

\textbf{Specification-to-code traceability and mining.}
Requirements traceability links natural-language requirements to source
artifacts using information retrieval, LSI, candidate-link generation, and deep
semantic representations~\cite{antoniol2002recovering,marcus2003recovering,hayes2006advancing,clelandhuang2014software,guo2017semantically}.
Specification mining infers likely properties from executions or recurring code
patterns~\cite{ernst2001dynamically,engler2001bugs,engler2001checking}.
These lines of work answer a different question: which artifacts are related,
or which properties are likely? \tool{} uses linking as an intermediate step
toward a stronger output: a conformance label with uncertainty notes,
implementation evidence, and optional validation. Because a wrong link can
become a false vulnerability report, \tool{} treats field identity, source
provenance, and missing evidence as part of the claim rather than metadata
outside the analysis.
